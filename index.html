문제 1. [C] 포인터와 배열 전달

C 난이도: 어려움 포인터 배열 포인터 산술 역참조 함수 호출 매개변수 전달

출제 개념: 포인터, 배열, 포인터 산술, 역참조, 함수 호출, 매개변수 전달, 반환값, 조건문, 반복문

출제 근거: 이전 파일 또는 핵심 개념 표에서 자주 등장한 개념 기반

문제:

다음 코드의 출력 결과를 작성하시오.

코드:

#include <stdio.h>

int calc(int *p, int n) {
    int sum = 0;

    for (int i = 0; i < n; i++) {
        if (*(p + i) % 2 == 0)
            sum += *(p + i);
        else
            sum -= *(p + i);

        *(p + i) += i;
    }

    return sum;
}

int main() {
    int a[5] = {3, 4, 5, 6, 7};
    int r = calc(a + 1, 3);

    printf("%d %d %d %d", r, a[1], a[2], a[3]);
}

답안 작성란:

앞뒤 공백과 줄 끝 공백은 무시하지만, 줄 순서와 대소문자는 구분합니다.

문제 1 정답 및 해설

정답

5 4 6 8

단계별 해설

  1. a 배열은 {3, 4, 5, 6, 7}입니다.
  2. calc(a + 1, 3)을 호출하므로 함수의 포인터 p는 a[1]을 가리킵니다.
  3. 따라서 함수 안에서 처리되는 값은 a[1], a[2], a[3]입니다.
  4. i=0: a[1]은 4이고 짝수이므로 sum은 4가 됩니다. a[1] += 0이므로 a[1]은 4입니다.
  5. i=1: a[2]는 5이고 홀수이므로 sum은 4 - 5 = -1이 됩니다. a[2] += 1이므로 a[2]는 6입니다.
  6. i=2: a[3]은 6이고 짝수이므로 sum은 -1 + 6 = 5가 됩니다. a[3] += 2이므로 a[3]은 8입니다.
  7. 함수의 반환값 r은 5이고, 최종 a[1], a[2], a[3]은 각각 4, 6, 8입니다.
  8. 헷갈릴 수 있는 부분: calc(a + 1, 3)은 배열 전체가 아니라 a[1]부터 3개만 처리합니다.

    int score;
};

int main() {

문제:

다음 코드의 출력 결과를 작성하시오.

코드:

#include <stdio.h>

int f(int a[][3], int r, int c) {
    int v = 0;

    for (int i = 0; i < r; i++) {
        for (int j = 0; j < c; j++) {
            if ((i + j) % 2 == 0)
                v += a[i][j];
            else
                v -= a[i][j];

            a[i][j] += v % 3;
        }
    }

    return v;
}

int main() {
    int x[2][3] = {
        {1, 2, 3},
        {4, 5, 6}
    };

    int r = f(x, 2, 3);

    printf("%d\n", r);

정답

-3 1 2 6

단계별 해설

  2. 변수 v는 0에서 시작합니다. (i + j) % 2 == 0이면 더하고, 아니면 뺍니다.
  3. 각 위치 처리 후 a[i][j] += v % 3을 수행합니다.

  9. i=1, j=2: v = 3 - 6 = -3, x[1][2]는 6 + 0 = 6입니다.
  10. 최종 r은 -3이고 출력되는 배열 값은 x[0][1]=1, x[1][0]=2, x[1][2]=6입니다.

#include <stdio.h>

int rec(char *s, int n) {
    if (n == 0 || *s == '\0')
        return 0;

    if (*s >= '0' && *s <= '9')
        return (*s - '0') + rec(s + 2, n - 1);

    return rec(s + 1, n - 1) - 1;
}

int main() {
    char str[] = "a1b23c4";

    printf("%d %c", rec(str, 5), *(str + 3));
}

문제 4 정답 및 해설

정답

5 2

단계별 해설

  1. 문자열 str은 a, 1, b, 2, 3, c, 4 순서입니다.
  2. rec(str, 5)를 호출합니다.
  3. 첫 문자는 'a'이고 숫자가 아니므로 rec(s + 1, 4) - 1입니다.
  4. 다음 문자는 '1'이고 숫자이므로 1 + rec(s + 2, 3)입니다. 여기서 s + 2는 문자 '2'를 가리킵니다.
  5. 문자 '2'는 숫자이므로 2 + rec(s + 2, 2)입니다. 여기서 s + 2는 문자 'c'를 가리킵니다.

  7. 문자 '4'는 숫자이므로 4 + rec(s + 2, 0)입니다.
  8. n이 0이 되면 0을 반환합니다.
  9. 거꾸로 계산하면 '4'는 4, 'c'는 4 - 1 = 3, '2'는 2 + 3 = 5, '1'은 1 + 5 = 6, 'a'는 6 - 1 = 5입니다.
  10. 따라서 rec(str, 5)는 5입니다. *(str + 3)은 인덱스 3의 문자이므로 '2'입니다.
  11. 헷갈릴 수 있는 부분: 숫자 문자를 만나면 s + 2로 이동하고, 숫자가 아니면 s + 1로 이동합니다.

문제 5 정답 및 해설

정답

3,16,13

단계별 해설

  1. 핵심은 필드 은닉과 메서드 오버라이딩의 차이입니다.
  2. A obj = new B();에서 객체는 B 객체로 생성되지만 참조 변수 타입은 A입니다.
  3. B 객체 생성 시 부모 클래스 A의 필드가 먼저 초기화되어 A.x = 1이 됩니다.
  4. A() 생성자 실행으로 x += 2가 수행되어 부모 쪽 x는 3이 됩니다.
  5. 이후 자식 클래스 B의 필드 x가 10으로 초기화됩니다.
  6. B() 생성자 실행으로 x += super.x가 수행됩니다. 여기서 x는 자식의 x이고 super.x는 부모의 x입니다.
  7. 따라서 B.x = 10 + 3 = 13입니다.
  8. obj.x는 필드 접근이므로 참조 변수 타입 A 기준입니다. 따라서 3입니다.
  9. obj.f()는 메서드 호출이므로 실제 객체 타입 B 기준으로 오버라이딩된 f()가 호출됩니다.
  10. B의 f()에서 x는 B.x=13, super.f()는 A.x=3을 반환하므로 13 + 3 = 16입니다.
  11. ((B)obj).x는 B 타입으로 형 변환했으므로 자식 필드 x인 13입니다.
  12. 헷갈릴 수 있는 부분: 필드는 참조 변수 타입을 따르고, 오버라이딩된 메서드는 실제 객체 타입을 따릅니다.

int mix(int *a, int n) {
    int r = 0;
    int *p = a + n - 1;

    for (int i = 0; i < n; i++) {
        r += *(p - i) - *(a + i);

        if (r % 2 == 0)
            *(a + i) += r;
        else
            *(p - i) -= i;
    }

    return r;
}

int main() {
    int arr[5] = {2, 5, 1, 4, 3};
    int result = mix(arr, 5);

    printf("%d\n", result);
    printf("%d %d %d %d", arr[0], arr[1], arr[3], arr[4]);

        arr[1] = arr[0];
        int r = work(arr);

    }
}

    }

        Unit u = new Beta();

        System.out.print(u.run(3) + " " + u.v);
    }

▼ 펼치기

모듈 내부의 요소들이 서로 관련되어 있는 밀집 정도를 나타냅니다.

결합도 (Coupling)

▼ 펼치기

🎯 응집도 (Cohesion) 계층 상세 요약 (높음 ➔ 낮음 순서)

1단계 기능적 응집도 (Functional Cohesion) - 가장 높음

모듈 내부의 모든 기능이 단일 목적을 수행하기 위해 구성된 경우입니다. 가장 이상적인 응집도입니다.

2단계 순차적 응집도 (Sequential Cohesion)

한 기능의 출력이 다른 기능의 입력으로 사용되는 순차적인 관계를 가진 기능들을 묶은 경우입니다. 때문에 기능 간의 의존성이 존재합니다.

3단계 통신적(교환적) 응집도 (Communicational Cohesion)

동일한 입력 데이터를 사용하거나 동일한 출력 데이터를 생성하는 기능들을 묶은 경우입니다.

4단계 절차적 응집도 (Procedural Cohesion)

모듈 내 기능들이 정해진 순서에 따라 실행되어야 할 때 이를 하나의 모듈로 묶은 경우입니다.

5단계 시간적 응집도 (Temporal Cohesion)

특정 시간에 함께 실행되어야 하는 기능들을 묶은 경우입니다.

6단계 논리적 응집도 (Logical Cohesion)

유사한 성격을 갖거나 특정 분류에 속하는 기능들을 하나의 모듈로 묶은 경우입니다.

7단계 우연적 응집도 (Accidental Cohesion) - 가장 낮음

모듈 내부의 구성요소들이 아무런 관련이 없는 경우입니다. 가장 낮은 수준의 응집도입니다.

🔗 결합도 (Coupling) 계층 상세 요약 (높음 ➔ 낮음 순서)

1단계 내용 결합도 (Content Coupling) - 가장 높음

한 모듈이 다른 모듈의 내부 기능이나 데이터를 직접 참조하거나 수정하는 경우입니다. 가장 높은 수준의 결합도입니다.

2단계 공통 결합도 (Common Coupling)

여러 모듈이 하나의 전역 변수나 공통 데이터 영역을 공유하여 사용하는 경우입니다.

3단계 외부 결합도 (External Coupling)

두 모듈이 외부에서 정의된 데이터 형식, 통신 프로토콜, 또는 특정 하드웨어를 공유하는 경우입니다.

4단계 제어 결합도 (Control Coupling)

한 모듈이 다른 모듈의 동작을 제어하는 제어 신호(Flag, Switch)를 전달하는 경우입니다.

5단계 스탬프 결합도 (Stamp Coupling)

두 모듈이 자료 구조(객체, 레코드 등)를 주고받지만, 그중 일부 데이터만 사용하는 경우입니다.

6단계 자료 결합도 (Data Coupling) - 가장 낮음

두 모듈이 필요한 최소한의 데이터(단순 파라미터)만을 주고받는 경우입니다. 가장 이상적인 결합도입니다.

데이터베이스 키 (Database Keys) 개요

데이터베이스 키는 튜플을 유일하게 식별하고 테이블 간 참조 무결성을 유지하기 위한 핵심 개념입니다. 실기에서는 후보키·기본키·외래키와 유일성/최소성 구분이 중요합니다.

🔑 데이터베이스 키 핵심 유형 총정리

키 종류 식별 조건 핵심 개념 정리
기본키 (Primary Key) 후보키 중 선택, Not Null 후보키 중 대표로 선택한 키입니다. 중복값과 Null 값을 가질 수 없습니다.
대체키 (Alternate Key) 후보키 중 나머지 후보키 중 기본키로 선택되지 않은 나머지 키입니다.
후보키 (Candidate Key) 유일성 O, 최소성 O 튜플을 유일하게 식별하는 최소 속성 집합입니다. 기본키의 후보가 됩니다.
슈퍼키 (Super Key) 유일성 O, 최소성 X 튜플을 유일하게 식별할 수 있지만 최소성은 만족하지 않을 수 있는 속성 집합입니다.
외래키 (Foreign Key) 다른 테이블의 기본키 참조 다른 릴레이션의 기본키를 참조하는 속성입니다. 테이블 간 관계와 참조 무결성을 표현합니다.

💡 키 지정의 두 가지 핵심 속성 (기출 암기 필수)

1. 유일성 (Uniqueness)

키 값 하나가 릴레이션의 튜플 하나만 식별해야 한다는 성질입니다. 즉 중복 불가입니다.

2. 최소성 (Minimality)

튜플 식별에 꼭 필요한 속성만으로 구성되어야 하는 성질입니다. 속성을 제거하면 유일성이 깨져야 합니다.

🛡️ 데이터베이스 무결성 규정 (Integrity Constraints) 총정리

🆔

개체 무결성 (Entity Integrity)

테이블의 기본키를 구성하는 어떤 속성도 Null 값이나 중복값을 가질 수 없음을 정의하는 규칙입니다.

🔗

참조 무결성 (Referential Integrity)

외래키 값은 참조하는 테이블의 기본키 값과 동일하거나 Null 값이어야 하며, 무효한 참조가 존재할 수 없음을 의미합니다.

데이터 모델 (Data Model) 개념 개요

데이터 모델은 현실 세계의 데이터를 데이터베이스 구조로 표현하기 위한 추상화 도구입니다. 실기에서는 릴레이션, 튜플, 속성, 차수, 카디널리티, 도메인, 스키마의 구분이 중요합니다.

📑 데이터 모델 구성요소 용어 해설

릴레이션 (Relation)

관계형 데이터베이스에서 테이블을 의미합니다. 행과 열로 구성됩니다.

튜플 (Tuple)

릴레이션의 행(Row)입니다. 레코드라고도 하며 하나의 개체 인스턴스를 나타냅니다.

속성 (Attribute)

릴레이션의 열(Column)입니다. 개체가 가지는 성질이나 항목을 나타냅니다.

카디널리티 (Cardinality / 기수)

릴레이션에 존재하는 튜플의 개수입니다. 데이터 추가·삭제에 따라 변합니다.

차수 (Degree / 열의 수)

릴레이션이 가진 속성의 개수입니다. 테이블 구조가 바뀌지 않으면 고정됩니다.

스키마 (Schema)

데이터베이스의 구조, 속성, 제약조건을 정의한 설계 정보입니다. 내포라고도 합니다.

도메인 (Domain)

하나의 특정한 속성(Attribute)이 취할 수 있는 동일한 데이터 타입의 원자적인 값들의 집합입니다. (예: 점수의 도메인은 0~100)

식별자 (Identifier)

개체(Entity) 내부에서 각 개체 인스턴스들을 고유하게 구분 및 구별하기 위해 기준이 되는 속성이며, 기본식별자, 보조식별자 등으로 구분됩니다.

인스턴스 (Instance)

설계된 데이터베이스의 틀(스키마)에 맞춰 실제 테이블에 들어온 실질적인 데이터 값들의 총합체(외연)를 뜻합니다.

정규화 (Normalization)

관계형 데이터베이스에서 중복을 줄이고 삽입·삭제·갱신 이상을 방지하기 위해 릴레이션을 분해하는 과정입니다.

반정규화 (Denormalization)

시스템의 조회(Query) 성능을 향상시키기 위해 의도적으로 정규화 규칙을 위배하고 중복, 통합, 분할을 수행하는 기법입니다. (과도할 경우 데이터 무결성이 저하됨)

1NF

제1정규형 (1NF): 도메인이 원자값

테이블의 모든 속성이 더 이상 분해할 수 없는 하나의 원자값(Atomic Value)으로만 구성되도록 정규화하는 단계입니다.

2NF

제2정규형 (2NF): 부분 함수적 종속 제거

기본키가 복합키(두 개 이상의 속성 조합)로 구성되어 있을 때, 기본키의 일부분에 종속되는 속성들을 분리해내어 완전 함수적 종속을 만족시키는 단계입니다.

3NF

제3정규형 (3NF): 이행적 함수 종속 제거

기본키를 거치지 않고 다른 속성을 거쳐 간접적으로 종속되는 관계인 이행적 함수 종속 (A → B 이고 B → C 이면 A → C 가 성립)을 제거하는 단계입니다.

BCNF

보이스-코드 정규형 (BCNF): 결정자이면서 후보키가 아닌 것 제거

모든 결정자가 후보키(Candidate Key)가 되도록 규정하는 강한 제3정규형 단계입니다. 후보키가 아닌 결정자 속성을 가진 종속 관계를 분해합니다.

4NF

제4정규형 (4NF): 다치 종속 제거

테이블 내에서 한 속성값에 따라 다른 속성들의 여러 값이 매핑되는 다치 종속 (A ↠ B) 관계를 제거하는 단계입니다.

5NF

제5정규형 (5NF / 조인 종속성 제거)

릴레이션의 모든 조인 종속성(Join Dependency)이 해당 릴레이션의 후보키를 통해서만 성립되도록 분할하는 최종 단계의 정규화입니다.

데이터베이스 설계 (Database Design) 프로세스

데이터베이스 설계는 사용자 요구사항을 분석해 개념적·논리적·물리적 구조로 단계적으로 구체화하는 과정입니다. 실기에서는 설계 단계와 산출물을 연결해 암기하는 것이 중요합니다.

🔄 데이터베이스 설계 4단계 절차 (요 ➔ 개 ➔ 논 ➔ 물)

1단계 : 요구사항 분석 (Requirements Analysis)

산출물 : 요구사항 명세서
핵심 활동: 사용자의 요구사항을 수집하고 분석하여 명세서 작성

사용자 요구를 수집·분석해 저장할 데이터와 처리 요구를 정리합니다. 산출물은 요구사항 명세서입니다.

2단계 : 개념적 설계 (Conceptual Design)

산출물 : 개념적 스키마, E-R 다이어그램
핵심 활동: 현실 세계를 추상적인 개념 모델로 표현

특정 DBMS에 종속되지 않는, 시스템의 핵심 개체(Entity)와 그들 간의 관계(Relationship)를 정의합니다. 주로 개체-관계 다이어그램(E-R Diagram)을 사용하여 표현합니다.

3단계 : 논리적 설계 (Logical Design)

산출물 : 논리적 스키마, 테이블 명세서, 논리적 데이터 모델 명세서
핵심 활동: 개념적 모델을 특정 DBMS가 지원하는 논리적 모델로 변환

개념적 설계 단계에서 만든 E-R 다이어그램을 실제 데이터베이스 모델(예: 관계형 모델)에 맞춰 테이블(릴레이션) 스키마로 변환합니다. 정규화 과정이 이 단계에서 수행됩니다.

4단계 : 물리적 설계 (Physical Design)

산출물 : 물리적 스키마, 테이블 정의서(물리적 스키마 정의서), 인덱스 정의서
핵심 활동: 논리적 모델을 실제 저장장치에 저장하기 위한 물리적 구조 정의

성능과 저장 효율을 고려해 파일 구조, 인덱스, 접근 경로, 저장 레코드 등을 결정합니다.

데이터 모델의 3요소 (3 Elements)

▼ 펼치기

데이터 모델 설계 시 기술되어야 하는 핵심 속성 3대 핵심 체계입니다.

📐 구조 (Structure) - 정적 성질

개체 타입들과 이들 간의 관계를 명세하여 데이터의 구조적 정적 배치 정의합니다.

⚙️ 연산 (Operations) - 동적 성질

데이터베이스에 표현된 개체 인스턴스를 처리하는 동적 작업(삽입, 삭제, 갱신 등)의 규정입니다.

🛡️ 제약 조건 (Constraints)

데이터베이스에 허용될 수 있는 실제 데이터의 논리적인 한계 및 허용 값의 조건 규칙을 명세합니다.

선

1. 결함 존재 증명: 테스트는 결함이 존재함을 증명할 뿐, 결함이 없음을 절대 증명하지 못함.

2. 완벽한 테스팅 불가능: 모든 시나리오와 조건의 값을 완벽히 대입해 테스팅하는 것은 불가능에 수렴함.

3. 초기에 테스팅 시작: 소프트웨어 생명 주기(SDLC)의 앞 단계(요구분석)부터 개입할수록 에러 수정 비용이 기하급수적으로 절감됨.

4. 결함 집중 (파레토 법칙): 시스템에 기인하는 대다수의 오류(약 80%)는 특정 극소수 모듈(약 20%)에 편중되어 발견됨.

5. 살충제 패러독스: 동일한 테스트 케이스를 장기간 반복하여 수행하면 새로운 결함을 절대 찾아낼 수 없음 (테스트 케이스의 정기적 교체 필요).

6. 테스팅은 정황에 의존: 비즈니스 도메인(모바일 앱 vs 원자력 시스템 등)의 특성에 맞추어 설계를 달리 행해야 함.

7. 오류-부재의 궤변: 사용자 요구사항 명세를 미충족한다면, 테스트에 결함이 검출되지 않았더라도 전혀 쓸모없는 소프트웨어임.

1. 처리량 (Throughput): 정해진 단위 시간 동안 애플리케이션이 완수하여 성공적으로 산출한 처리 작업의 양 (지표가 클수록 고성능).

2. 응답 시간 (Response Time): 사용자가 시스템에 임의 요청을 전송한 시점부터 최초의 일부 응답 데이터가 브라우저에 도달하는 순간까지 걸린 물리 시간 (지표가 짧을수록 우수).

3. 경과 시간 / 반환 시간 (Turnaround Time): 사용자가 배치성 요구 작업이나 연산을 시스템에 전송 완료하여 완전히 처리된 총 결괏값을 온전히 돌려받을 때까지 경과한 총체적인 대기 시간 (지표가 짧을수록 우수).

4. 자원 사용률 (Resource Usage): 통신 및 작업 트래픽 처리 중 시스템에 위치한 CPU, 메모리, 영구 디스크, 네트워크 대역폭 등의 소모 비율 (동일 연산 대비 낮을수록 자원 안정성 탁월).

개념 및 가치 정리:

응용 프로그램의 코드 변경(새로운 모듈 삽입, 패치 수정, 버그 교정 등)이 가해진 후, 해당 수정사항이 기존에 정상적으로 구동하던 비영향권 타 부문 시스템 구역에 예기치 못한 사이드 이펙트(Side Effect) 결함이나 연쇄 오작동을 야기하지 않았는지 확인하기 위해 이전에 성공했던 기존 테스트 케이스를 똑같이 다시 반복 실행하는 점검 테스팅입니다.

데이터베이스 키 스키마 갱신, 프로토콜 포트 변경, 또는 신규 디자인 패턴을 도입하는 소스코드 수정이 일어날 때마다 무조건 수행되어야 무결성을 사수할 수 있습니다.

테스트 커버리지 (Test Coverage) 개요

테스트 커버리지는 테스트 케이스가 소스코드의 구조를 얼마나 실행했는지 나타내는 지표입니다. 실기에서는 구문, 결정, 조건, 조건/결정, MC/DC 차이를 구분합니다.

🎯 화이트박스 테스트 검증 기준 (테스트 커버리지 유형)

구문

구문(문장) 커버리지 (Statement Coverage)

요약 키워드: 모든 명령문, 최소 한 번 이상 실행

모든 명령문이 최소 한 번 이상 실행되도록 테스트하는 가장 기본 기준입니다.

결정

결정(분기) 커버리지 (Decision / Branch Coverage)

요약 키워드: 각 결정문, 참(True)/거짓(False), 한 번 이상 수행

각 결정문 결과가 참과 거짓을 모두 한 번 이상 갖도록 테스트하는 기준입니다.

조건

조건 커버리지 (Condition Coverage)

요약 키워드: 결정문 내 개별 조건식 각각 참(True)/거짓(False), 한 번 이상 수행

전체 결정문의 결과와 상관없이, 결정문 내부에 결합되어 있는 개별 조건식(Condition)들이 각각 참(True)과 거짓(False)의 결과값을 모두 한 번 이상 갖도록 테스트를 수립합니다.

조/결

조건/결정 커버리지 (Condition / Decision Coverage)

요약 키워드: 조건 커버리지 + 결정 커버리지 조건 동시 충족

개별 조건식들이 각각 참/거짓을 한 번 이상 만족하도록 할 뿐만 아니라, 그 조건식들이 조립된 전체 결정문의 최종 연산 결과 역시 참/거짓 결과를 골고루 한 번씩 유발하도록 상호 보완 설계하는 강력한 기준입니다.

변경

변경 조건/결정 커버리지 (MC/DC - Modified Condition / Decision Coverage)

요약 키워드: 개별 조건식이 전체 결과에 독립적으로 고유한 영향을 주도록 구성

다른 조건식들의 영향력을 정적으로 묶어 고정해둔 채로, 오직 특정 한 개의 개별 조건식 값의 변화(참 ↔ 거짓)만으로 전체 결정문의 최종 연산 결과를 반전(참 ↔ 거짓)시키도록 검증함으로써 소스코드의 안전무결성을 철저히 증명하는 고급 기준입니다.

UML 다이어그램 (Unified Modeling Language Diagram) 개요

UML 다이어그램은 소프트웨어의 요구사항, 구조, 동작을 그림으로 표현하는 객체지향 모델링 표준 언어입니다. 정보처리기사 실기에서는 다이어그램의 분류와 관계 기호/의미를 구분하는 문제가 자주 출제됩니다.

UML 다이어그램 개요

▼ 펼치기

객체지향 시스템을 시각적으로 표현하는 표준 모델링 도구입니다.

구조 다이어그램

▼ 펼치기

시스템의 정적인 구성 요소와 관계를 표현합니다.

행위 다이어그램

▼ 펼치기

시스템의 동적인 동작, 흐름, 상호작용을 표현합니다.

UML 다이어그램 관계

▼ 펼치기

클래스, 객체, 컴포넌트 사이의 연결 의미를 표현합니다.

📌 UML 다이어그램 개요

UML은 Unified Modeling Language의 약자로, 소프트웨어 분석·설계 결과를 표준화된 그림으로 표현합니다. 실기에서는 주로 구조 다이어그램, 행위 다이어그램, 관계 종류를 구분하는 문제가 중요합니다.

🧱 구조 다이어그램 (Structure Diagram)

시스템을 구성하는 요소와 관계처럼 시간에 따라 변하지 않는 정적 구조를 표현합니다.

클래스 다이어그램: 클래스, 속성, 연산, 관계 표현
객체 다이어그램: 특정 시점의 객체와 객체 관계 표현
컴포넌트 다이어그램: 컴포넌트와 인터페이스 의존 관계 표현
배치 다이어그램: 노드, 장치, 실행 환경의 물리적 배치 표현
복합체 구조 다이어그램: 내부 구조와 구성 요소의 협력 관계 표현
패키지 다이어그램: 패키지 간 포함·의존 관계 표현

🏃 행위 다이어그램 (Behavior Diagram)

기능 수행 과정, 상태 변화, 객체 간 메시지 흐름처럼 시간에 따라 변하는 동적 행위를 표현합니다.

유스케이스 다이어그램: 사용자와 시스템 기능의 관계 표현
시퀀스 다이어그램: 시간 순서에 따른 객체 메시지 표현
커뮤니케이션 다이어그램: 객체 간 메시지와 연결 관계 표현
상태 다이어그램: 객체의 상태 변화와 전이 표현
활동 다이어그램: 작업 처리 흐름과 제어 흐름 표현
타이밍 다이어그램: 시간 제약에 따른 상태 변화 표현

🔗 UML 다이어그램 관계

연관 관계: 두 클래스가 서로 관련되어 있음을 표현
집합 관계: 전체와 부분 관계, 부분이 독립적으로 존재 가능
포함 관계: 강한 전체·부분 관계, 전체가 사라지면 부분도 소멸
일반화 관계: 상위 클래스와 하위 클래스의 상속 관계
의존 관계: 한 요소의 변경이 다른 요소에 영향을 주는 관계
실체화 관계: 인터페이스의 기능을 클래스가 실제로 구현하는 관계

🧠 핵심 암기 포인트

구조 vs 행위

구조 다이어그램은 정적 구조, 행위 다이어그램은 동적 행위를 표현합니다.
